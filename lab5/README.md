## 1. Початок роботи

1. Інсталяція необхідних пакетів:
   ```bash
   npm install
   ```

2. Запуск середовища Expo:
   ```bash
   npx expo start
   ```

## 2. Ключові особливості проекту

- **Управління доступом**: Використання глобального `AuthContext` для моніторингу статусу користувача.
- **Приватні маршрути**: Реалізовано механізм захисту контенту в групі `(app)`, що автоматично спрямовує неавторизованих гостей на сторінку логіну.
- **Інтерфейс магазину**: Інтерактивна вітрина товарів на базі `FlatList` з детальною інформацією про кожен юніт.
- **Гнучка маршрутизація**: Динамічні сторінки товарів на основі шаблону `[id].jsx`.
- **Окремі зони додатку**: Чіткий поділ на авторизаційний блок `(auth)` та основний функціонал.
- **Навігаційна стійкість**: Налаштована сторінка помилки `+not-found.jsx` для некоректних посилань.

## 3. Відповіді на контрольні питання

### 1. Яким чином за допомогою Expo Router реалізується перенаправлення неавторизованого користувача?
В Expo Router це  реалізується через перевірку стану авторизації (наприклад, у layout або guard-компоненті). Якщо користувач не авторизований — викликається router.replace('/login'), щоб примусово перекинути його на сторінку входу.
### 2. У чому полягає різниця між використанням компонента <Link> та метода router.push()?
- **Link**: Це декларативний спосіб навігації. Він схожий на тег `<a>` у вебі, підтримує доступність (accessibility) та дозволяє легко обгортати компоненти за допомогою `asChild`. Краще підходить для статичних посилань у UI.
- **router.push**: Це імперативний спосіб навігації через JavaScript код. Використовується всередині обробників подій (наприклад, після успішного запиту до API або валідації форми). Надає більше контролю над програмною логікою переходу.

### 3. Як працюють динамічні маршрути в Expo Router і як отримати передані параметри?
Динамічні маршрути створюються шляхом використання квадратних дужок у назві файлу, наприклад `[id].jsx`. Expo Router автоматично зчитує частину URL, що відповідає цьому місцю, і передає її як параметр. Отримати цей параметр можна за допомогою хука `useLocalSearchParams()`, який повертає об'єкт з ключем `id` (або іншою назвою, вказаною у дужках).

### 4. Чому стан авторизації доцільно зберігати у глобальному контексті (React Context), а не в локальному стані компонента?
`Context API` дозволяє передавати дані (стан авторизації, дані користувача, функції login/logout) через дерево компонентів без необхідності передавати пропси вручну на кожному рівні (prop drilling). Оскільки стан авторизації потрібен майже в кожній частині додатку (для логіки редиректів, відображення профілю тощо), Context є ідеальним інструментом для централізованого керування цим станом.

### 5. Для чого використовуються групи маршрутів (folderName) і як вони впливають на URL-адресу?
Групи маршрутів (папки в дужках, наприклад `(auth)` та `(app)`) використовуються для логічної організації структури проєкту без впливу на URL-адресу. Вони дозволяють:
- Мати різні кореневі Layout-файли для різних частин додатку.
- Легко застосовувати спільну логіку (наприклад, перевірку авторизації) до всієї групи екранів.
- Робити структуру файлів зрозумілішою та масштабованою.
